PineTree API

PineTree API lets developers create checkout sessions, retrieve payment status, receive signed webhook events, and reconcile payments across enabled payment rails.

PineTree API uses versioned REST endpoints. The current API path prefix is /api/v1.

Early Access

PineTree Payments is currently in early access. API access, SDK packages, and hosted checkout are available to approved merchants. To request access, contact info@pinetree-payments.com or through your PineTree merchant account.

Core Flow

  1. Create a Checkout Session from your server.
  2. Redirect the customer to checkoutUrl.
  3. The customer completes payment on an enabled rail.
  4. PineTree sends a signed webhook event.
  5. Fulfill the order after payment.confirmed.

Quickstart

Create a checkout session server-side, redirect the customer, then fulfill only after a confirmed webhook.

Install the Node SDK

npm install @pinetreepayments/node

Create a Checkout Session

import { PineTree } from "@pinetreepayments/node"

const pinetree = new PineTree(process.env.PINETREE_API_KEY)

const session = await pinetree.checkout.sessions.create({
  amount: 2500,
  currency: "USD",
  reference: "order_1042",
  successUrl: "https://yoursite.com/paid",
  cancelUrl: "https://yoursite.com/cancel"
})

res.redirect(session.checkoutUrl)

Verify Webhooks Before Fulfillment

app.post("/webhooks/pinetree", express.raw({ type: "application/json" }), (req, res) => {
  const event = pinetree.webhooks.constructEvent(
    req.body,
    req.headers,
    process.env.PINETREE_WEBHOOK_SECRET
  )

  if (event.type === "payment.confirmed") {
    fulfillOrder(event.data.object.reference)
  }

  res.json({ received: true })
})

Authentication

Use secret keys on your server. Use browser keys only for public checkout creation. Never expose secret API keys in frontend code.

CredentialWhere to Use
pt_live_*Server REST API calls and backend SDK usage.
pk_live_*Browser checkout creation only.

Required Headers

Authorization: Bearer pt_live_your_api_key_here
Content-Type: application/json

The Idempotency-Key header is optional. Include it on POST /api/v1/checkout/sessions to prevent duplicate sessions on retries. See Idempotency.

Checkout Sessions

Checkout Sessions create hosted payment pages. Create sessions from your server and fulfill orders only after confirmed webhook events.

Endpoints

POST/api/v1/checkout/sessions

Creates a hosted Checkout Session. Requires checkout.sessions:create.

GET/api/v1/checkout/sessions

Lists Checkout Sessions. Supports status, reference, limit, date, and cursor filters. Requires checkout.sessions:read.

GET/api/v1/checkout/sessions/{id}

Retrieves a Checkout Session by ID. Requires checkout.sessions:read.

POST/api/v1/checkout/sessions/{id}/cancel

Cancels an open Checkout Session. Requires checkout.sessions:write.

POST/api/v1/checkout/sessions/{id}/expire

Expires an open Checkout Session. Requires checkout.sessions:write.

POST/api/v1/browser/checkout/sessions

Creates a browser checkout session using a public key (X-PineTree-Public-Key).

Payments

Use Payments for reconciliation, status lookup, and back-office workflows.

GET/api/v1/payments/{id}

Retrieves a Payment object. Requires payments:read.

Payment Status Values

The Payment object returns status: "paid" when a payment is confirmed. The Confirmed state in webhook events maps to status: "paid" in the API response.

Always use the payment.confirmed webhook event to trigger fulfillment. Do not poll payment status as your primary fulfillment signal.

API statusMeaning
openPayment created, awaiting funds
processingPayment detected on-chain
paidPayment confirmed — fulfill now
failedPayment failed
expiredSession timed out
canceledCanceled by merchant

Rails & Assets

Rails are payment paths. Assets are what the customer pays on that path.

RailAssets / CurrencyDescription
solanaSOL, USDCSolana Pay payments.
baseETH, USDCBase network payments.
bitcoin_lightningBTCLightning invoice payments.
shift4Card / USDCard payments for approved merchants. Contact support to enable.

Card processing via Shift4 is available to approved merchants through the PineTree merchant onboarding flow. Contact info@pinetree-payments.com to apply. Additional card processors are available in early access.

Invalid rail examples

Do not combine the rail and asset into a single compound value. Use the rail field for the network path and the asset field for what the customer pays. PineTree returns both fields separately on every payment and webhook event.

Payment States

Confirmed is the successful visible payment state. Failed, Expired, and Incomplete are separate terminal outcomes. The Payment API object uses status: "paid" for the confirmed state.

StateMeaningTerminalColor
WaitingPayment request open, no funds detectedNoBlue
ProcessingPayment detected, awaiting final confirmationNoBlue
ConfirmedPayment completedYesGreen
FailedProvider, network, or payment attempt failedYesRed
ExpiredPayment window timed outYesRed
IncompleteCustomer abandoned or backed out or no funds were sentYesRed

Webhooks

Webhook deliveries are signed with PineTree headers. Verify the raw body before processing.

HeaderDescription
PineTree-SignatureHMAC-SHA256 signature of PineTree-Timestamp + '.' + the raw request body.
PineTree-TimestampISO 8601 timestamp for replay protection. Must be within 5 minutes.
PineTree-Event-IdUnique event ID for deduplication.
PineTree-Event-Schemapayments-v1

Verification

// Use express.raw() — do not JSON.parse() before verification
const event = pinetree.webhooks.constructEvent(
  req.body,
  req.headers,
  process.env.PINETREE_WEBHOOK_SECRET
)

The signature is computed as HMAC-SHA256(PineTree-Timestamp + "." + rawBody). PineTree rejects events with a timestamp older than 5 minutes to prevent replay attacks.

Idempotency

Use idempotency keys to safely retry Checkout Session creation without duplicate sessions. Add the Idempotency-Key header to POST /api/v1/checkout/sessions. 

const session = await pinetree.checkout.sessions.create(
  { amount: 2500, reference: "order_1042" },
  { idempotencyKey: "order_1042" }
)
ScenarioResult
Same key and same bodyReturns original session.
Same key and different bodyReturns idempotency_key_conflict.
Request still in progressReturns idempotency_request_in_progress.

Go Live

Complete these checks before accepting real payments.

AreaChecklist
API keysSecret key created with minimal permissions and stored server-side. No pt_live_* keys in frontend code.
WebhooksHTTPS endpoint configured. Raw-body verification enabled. PineTree-Signature verified on every event. Duplicate events handled via eventId.
Payment flowsConfirmed payments fulfill orders. Failed, Expired, and Incomplete payments do not fulfill. payment.confirmed is the fulfillment trigger.
RailsEnabled payment rails are configured and tested with small live amounts. At least one crypto rail active.

Not yet supported

The following are not currently available through the PineTree API:

  • Refund API (refunds processed through the merchant dashboard)
  • Payout or settlement API (settlement configured in dashboard settings)
  • Dispute management API
  • Sandbox or test-mode keys (PineTree uses live keys only)
  • Stripe and Fluid Pay card processing (available to early-access merchants on request)
