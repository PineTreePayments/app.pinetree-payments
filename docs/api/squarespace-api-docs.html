PineTree API

Accept crypto payments across Solana, Base, Lightning, and more. The PineTree API gives you checkout sessions, real-time webhooks, and a hosted payment page — so your customers can pay with any wallet in under 60 seconds.

Core concepts

Checkout Session

A payment intent with a hosted checkout URL. Customers select a network and complete payment in the PineTree checkout. Sessions expire after 24 hours.

Payment

Created when a customer initiates a payment attempt. Tracks on-chain status, network, rail, and amount. Transitions through: open → processing → paid | failed | incomplete.

Webhook Event

Signed HTTP POST delivered to your endpoint when payment status changes. Use payment.confirmed to fulfill orders. Verified with HMAC-SHA256.

Quickstart

Integrate PineTree in minutes. Create a session on your server, redirect your customer, and listen for the confirmation webhook.

1
Get an API key

Go to Developer → API Keys in your dashboard. Create a key with checkout.sessions:create permission. Store it as PINETREE_API_KEY.

2
Install the Node SDK
npm install @pinetreepayments/node
3
Create a checkout session
import { PineTree } from "@pinetreepayments/node"

const pinetree = new PineTree(process.env.PINETREE_API_KEY!)

const session = await pinetree.checkout.sessions.create({
  amount: 2500,                            // $25.00 in cents
  currency: "USD",
  reference: "order_1042",
  successUrl: "https://yoursite.com/success",
  cancelUrl: "https://yoursite.com/cancel",
})

// Redirect the customer
res.redirect(session.checkoutUrl)
4
Verify the webhook
// Express — must use express.raw() before parsing
app.post("/webhooks/pinetree",
  express.raw({ type: "application/json" }),
  (req, res) => {
    const event = pinetree.webhooks.constructEvent(
      req.body,
      req.headers,
      process.env.PINETREE_WEBHOOK_SECRET!
    )
    if (event.type === "payment.confirmed") {
      fulfillOrder(event.data.object.reference)
    }
    res.json({ received: true })
  }
)
No sandbox mode. PineTree uses live keys only. Use small amounts (e.g., $0.01) for integration testing.

Authentication

PineTree uses API keys for server-to-server auth and public keys for browser-side checkout. Never expose secret keys in frontend code.

Secret API keys

Created in Developer → API Keys. Format: pt_live_<64 hex chars>

Authorization: Bearer pt_live_your_api_key_here
Content-Type: application/json

Key permissions

PermissionGrants
checkout.sessions:createCreate checkout sessions
checkout.sessions:readList and retrieve sessions
payments:readRetrieve payment objects
webhooks:readList webhook deliveries
webhooks:writeRetry webhook deliveries

Auth errors

CodeStatusCause
missing_api_key401No Authorization header
invalid_api_key401Invalid format or revoked key
missing_permission403Key lacks required scope
Security: Never use pt_live_* keys in browser code. Use pk_live_* public keys with the Browser SDK for frontend checkout flows.

Checkout Sessions

A checkout session is a single payment intent. Create one server-side, redirect the customer to checkoutUrl, and receive a webhook when payment is confirmed.

POST /api/v1/checkout/sessions

Create a checkout session. Requires checkout.sessions:create.

curl -X POST https://app.pinetree-payments.com/api/v1/checkout/sessions \
  -H "Authorization: Bearer pt_live_..." \
  -H "Content-Type: application/json" \
  -H "Idempotency-Key: order_1042" \
  -d '{
    "amount": 2500,
    "currency": "USD",
    "reference": "order_1042",
    "customer": { "email": "jane@example.com" },
    "successUrl": "https://yoursite.com/success",
    "cancelUrl": "https://yoursite.com/cancel"
  }'
GET /api/v1/checkout/sessions

List sessions. Supports status, reference, limit, starting_after, date filters.

GET /api/v1/checkout/sessions/{id}

Retrieve a single session by ID.

POST /api/v1/checkout/sessions/{id}/cancel

Cancel an open session.

POST /api/v1/checkout/sessions/{id}/expire

Immediately expire a session.

Payments

A payment object is created when a customer initiates a payment attempt. It tracks on-chain status, network, and amount.

GET /api/v1/payments/{id}

Retrieve a payment. Requires payments:read.

{
  "id": "pay_01abc...",
  "object": "payment",
  "status": "paid",
  "amount": 2500,
  "currency": "USD",
  "network": "solana",
  "rail": "solana",
  "reference": "order_1042",
  "metadata": {},
  "createdAt": "2026-06-16T12:01:00.000Z",
  "updatedAt": "2026-06-16T12:02:30.000Z"
}

Payment lifecycle

CREATED → PENDING → PROCESSING → CONFIRMED (status: "paid")
                             └→ FAILED    (status: "failed")
              └→ INCOMPLETE               (status: "incomplete")

Terminal states (paid, failed, incomplete) are permanent. Use the payment.confirmed webhook event for order fulfillment.

Webhooks

PineTree delivers HMAC-signed events to your endpoint when payment status changes. Always verify the signature before processing.

Webhook headers

HeaderDescription
PineTree-SignatureHMAC-SHA256 of the raw body using your signing secret
PineTree-TimestampISO 8601 — must be within 5 minutes of current time
PineTree-Event-IdUnique event ID for deduplication
PineTree-Webhook-Version2026-06-12

Signature verification

// Node SDK (recommended)
const event = pinetree.webhooks.constructEvent(
  rawBody,          // Buffer — do not JSON.parse() first
  req.headers,      // pass the full headers object
  process.env.PINETREE_WEBHOOK_SECRET!
)

// Manual verification (Node.js)
import { createHmac, timingSafeEqual } from "node:crypto"

function verify(rawBody, signature, timestamp, secret) {
  const age = Math.abs(Date.now() - new Date(timestamp).getTime()) / 1000
  if (age > 300) return false  // reject stale events

  const expected = createHmac("sha256", secret)
    .update(rawBody).digest("hex")
  const actual = signature.replace(/^sha256=/i, "")

  return timingSafeEqual(
    Buffer.from(expected, "hex"),
    Buffer.from(actual, "hex")
  )
}

Event types

EventWhen it fires
payment.createdPayment object first created
payment.pendingCustomer wallet action detected
payment.processingTransaction broadcast; awaiting confirmation

Register your webhook endpoint and copy the signing secret from Developer → Webhooks in your dashboard. Use the signing secret as PINETREE_WEBHOOK_SECRET.

SDKs & Libraries

PineTree provides official SDKs for Node.js, browser JavaScript, and React. The REST API is available without an SDK.

Node SDK

@pinetreepayments/node — for server-side session creation, payment retrieval, and webhook verification.

npm install @pinetreepayments/node
import { PineTree } from "@pinetreepayments/node"
const pinetree = new PineTree(process.env.PINETREE_API_KEY!)

const session = await pinetree.checkout.sessions.create({ amount: 2500 })
const payment = await pinetree.payments.retrieve("pay_01abc...")
await pinetree.webhookDeliveries.retry("wdel_01abc...")

Browser JavaScript SDK

@pinetreepayments/js — open checkout from browser code using a public key. Supports redirect, popup, and embedded modes.

npm install @pinetreepayments/js
import { PineTree } from "@pinetreepayments/js"
const pinetree = new PineTree("pk_live_your_public_key")

const result = await pinetree.checkout.open({
  amount: 2500,
  currency: "USD",
  mode: "popup",        // "redirect" | "popup" | "embedded"
})

result.on("complete", (event) => {
  console.log("Paid:", event.sessionId)
})

React SDK

@pinetreepayments/react — React components for checkout buttons and embedded checkout.

npm install @pinetreepayments/react

Go-Live Checklist

Complete before accepting real payments.

API & Keys

  • Secret API key created with minimal required permissions
  • API key stored in environment variables, not source code
  • No pt_live_* keys in frontend code
  • Public browser key created if using Browser SDK

Webhooks

  • Webhook endpoint registered (HTTPS only)
  • Signing secret stored as PINETREE_WEBHOOK_SECRET
  • PineTree-Signature verified on every event
  • Duplicate events handled via eventId deduplication
  • Events processed asynchronously — handler returns 2xx immediately

Payment flows tested

  • Happy path: session created → paid → order fulfilled
  • Failed payment: not fulfilled, customer notified
  • Expired/incomplete: not fulfilled
  • Canceled session: not fulfilled

Provider & environment

  • At least one payment rail configured in the dashboard
  • CHECKOUT_SESSION_SECRET, TERMINAL_SESSION_SECRET set in production
  • Treasury wallet addresses verified
  • Reports reviewed in Dashboard → Reports

Provider Integrations

For payment processors and infrastructure partners who want to integrate with PineTree. Every provider implements a standard adapter interface.

Adapter interface

interface ProviderAdapter {
  createPayment(input): Promise<CreatePaymentResult>
  getPaymentStatus(id): Promise<"pending"|"paid"|"failed"|"expired">
  verifyWebhook(rawBody, headers): boolean  // must use timingSafeEqual
  translateEvent(event): NormalizedEvent    // map to payment.confirmed etc.
  healthCheck(): Promise<{ status, message }>
}

What PineTree needs from a new provider

  • REST API docs + sandbox credentials
  • Webhook event spec (types, payload schema, signing algorithm)
  • HMAC signing secret for webhook verification
  • Settlement model — direct to merchant wallet or custody?
  • Fee model — flat, percentage, or hybrid?
  • Refund API — supported?
  • Rate limits and uptime SLA
  • Test payment flow (success, failure, expiry in sandbox)
